# HTML 可玩广告总需求文档

本文档是本项目中所有 HTML 可玩广告的**总规范**。所有后续交付成果必须严格遵守这些标准。

## 1. 游戏流程与开始条件

- **禁止自动开始**：加载完成后，游戏逻辑（计时器、怪物生成、移动等）**绝不能**自动开始。
- **首次交互触发**：游戏必须保持在“准备就绪”状态，直到玩家在屏幕上执行第一次交互（点击/触摸）。
- **待机状态**：在第一次交互之前，游戏应显示初始场景/UI 和教程提示，等待输入。

## 2. 摇杆控制 (Joystick Control)

### 2.1. 状态 A: 待机/引导阶段 (Idle/Tutorial Phase)

- **视觉位置**：屏幕**右下角**固定显示摇杆 UI。
- **目的**：配合教程手势 (Hand Animation) 提示用户操作位置。

### 2.2. 状态 B: 游戏进行中 (Gameplay Phase)

- **动态跟随 (Dynamic Positioning)**: 摇杆不再固定。
- **触发逻辑**：
  - 当用户在屏幕**任意位置**按下/触摸时，摇杆的中心点 (Base) 应**立即跳转**到该触摸位置。
  - **隐显规则**: 触摸开始 (TouchStart) 时显示，触摸结束 (TouchEnd) 时隐藏（或淡出/复位）。
- **输入逻辑**: 全屏触控有效。摇杆的移动量基于用户手指相对于**按下点**的位移。

## 3. 教程与引导

每次游戏启动时，必须显示强制性教程，指导玩家如何移动。

- **视觉提示**：一个**箭头**指向摇杆的视觉位置（右下角）。
- **动画**：箭头应有动画效果（例如上下浮动或滑动）以吸引注意力。
- **时机**：在“待机状态”下一加载立即显示。
- **消失条件**：一旦玩家成功执行第一次移动输入，教程手势/箭头应消失。

## 4. 技术限制

- **格式**：单个 HTML 文件。
- **方向**：针对竖屏（Portrait）移动端游玩进行优化（除非另有说明，但“右下角摇杆”暗示标准的移动端竖屏/横屏）。
- **兼容性**：必须在移动端 Webview（iOS/Android）上流畅运行。
